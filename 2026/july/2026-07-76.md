## 📝 Core Concepts (Quick Recap)

**Logical execution order (memorize this — asked constantly):**
```
1. FROM        → get the table(s)
2. WHERE       → filter individual rows (before grouping)
3. GROUP BY    → group filtered rows
4. HAVING      → filter groups (after aggregation)
5. SELECT      → pick/compute columns
6. ORDER BY    → sort final result
```
This is why a SELECT alias can't be used in WHERE (WHERE runs before SELECT), but CAN be used in ORDER BY (runs after SELECT).

**WHERE vs HAVING:**
| | WHERE | HAVING |
|---|---|---|
| Filters | Individual rows | Groups (after aggregation) |
| Runs | Before GROUP BY | After GROUP BY |
| Can use aggregates? | ❌ No | ✅ Yes |
| Use for | Row-level conditions (e.g. `Status='Delivered'`) | Aggregate conditions (e.g. `SUM(Amount)>1000`) |

**GROUP BY rule:** Every column in SELECT that is NOT wrapped in an aggregate function MUST appear in GROUP BY. Otherwise SQL doesn't know which value to display per group.

**NULL handling:**
- `NULL` ≠ anything, even another NULL → use `IS NULL` / `IS NOT NULL`, never `= NULL`
- `SUM`, `AVG`, `COUNT(column)` ignore NULLs automatically
- `COUNT(*)` counts all rows regardless of NULLs; `COUNT(column)` counts only non-NULL values in that column

**DISTINCT vs GROUP BY:**
- DISTINCT = remove duplicates, no aggregation
- GROUP BY = group rows so aggregate functions (SUM/COUNT/AVG) can compute per group
- If you need a calculation alongside unique values → GROUP BY. If you just need unique values → DISTINCT.

## 📚 Full 25 Interview Questions + Detailed Answers (Reference Bank)

### 🟢 Beginner

**1. What is the logical execution order of a SQL query?**
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. Explains why a SELECT alias can't be used in WHERE (runs before SELECT) but can be used in ORDER BY (runs after SELECT).
*Mistake:* reciting written order (SELECT first) instead of execution order.

**2. Difference between WHERE and HAVING?**
WHERE filters individual rows before grouping and cannot use aggregates. HAVING filters groups after aggregation, used when the condition involves SUM/COUNT/AVG etc.
*Mistake:* using HAVING for a plain row filter — slower, since WHERE reduces rows before the costlier GROUP BY step.

**3. What does GROUP BY do?**
Collapses rows sharing the same value(s) in specified columns into groups, so aggregate functions compute per group instead of over the whole table.

**4. Difference between COUNT(*) and COUNT(column)?**
COUNT(*) counts every row including NULLs. COUNT(column) counts only non-NULL values in that column. They diverge the moment the column has NULLs.

**5. What is the default sort order of ORDER BY?**
Ascending (ASC) if not specified.

**6. Can you use WHERE with aggregate functions? Why/why not?**
No — WHERE executes before GROUP BY/aggregation happens, so aggregate values don't exist yet at that stage. Aggregate-based conditions must go in HAVING.

**7. What does DISTINCT do?**
Removes duplicate rows from the result set based on all selected columns.

**8. How do you filter for NULL values?**
`WHERE column IS NULL` / `IS NOT NULL` — never `= NULL`, since NULL isn't equal to anything, including itself.

**9. Difference between `=` and `IS` in SQL?**
`=` is standard equality for actual values. `IS` is specifically for NULL comparisons, because `=` can never evaluate true against NULL.

### 🟡 Intermediate

**10. Why can't you use a SELECT alias inside WHERE?**
Because WHERE executes before SELECT in the logical order — the alias doesn't exist yet when WHERE runs. (Can be used in ORDER BY since that runs after SELECT.)

**11. Every non-aggregated column in SELECT must appear where?**
In the GROUP BY clause. Otherwise SQL can't determine which value to show for that column per group.

**12. What happens if you GROUP BY a column but don't aggregate any other column?**
If another SELECT column isn't aggregated and isn't in GROUP BY, most databases (SQL Server, PostgreSQL, MySQL strict mode) throw an error — SQL can't decide which value to display since the group may contain multiple different values for that column.

**13. How do NULLs behave inside SUM, AVG, COUNT?**
SUM, AVG, and COUNT(column) all ignore/skip NULL values automatically — they don't treat NULL as zero. COUNT(*) counts rows regardless of NULLs anywhere.

**14. Can you filter on multiple aggregate conditions in HAVING?**
Yes — combine with AND/OR, e.g. `HAVING SUM(Amount) > 100000 AND COUNT(*) > 10`.

**15. Difference between WHERE and HAVING when both could technically filter data?**
WHERE should always be preferred for row-level conditions since it filters before the expensive GROUP BY step, reducing the data aggregation has to process. HAVING should be reserved only for conditions that genuinely require an aggregate value.

**16. Can ORDER BY use a column that isn't in SELECT?**
Yes, in most databases — ORDER BY can reference any column from the underlying table(s), not just ones in SELECT, since it operates on the full row before final projection in some engines (behavior can vary slightly by RDBMS).

**17. Difference between DISTINCT and GROUP BY when producing unique values?**
DISTINCT simply removes duplicate rows, no aggregation. GROUP BY groups rows to allow aggregate calculations per group; producing unique values is a side effect. Use GROUP BY the moment you need any SUM/COUNT/AVG alongside the unique values.

**18. How would you find duplicate rows using GROUP BY + HAVING?**
```sql
SELECT Email, COUNT(*) 
FROM Customers 
GROUP BY Email 
HAVING COUNT(*) > 1;
```
Group by the column(s) that define a "duplicate," then keep only groups with more than 1 occurrence.

### 🔴 Advanced

**19. Find the 2nd highest value without LIMIT/TOP or MAX subquery.**
```sql
SELECT MIN(Salary) FROM 
(SELECT DISTINCT TOP 2 Salary FROM Employees ORDER BY Salary DESC) AS Top2;
```
Or using window functions: `SELECT Salary FROM (SELECT Salary, DENSE_RANK() OVER (ORDER BY Salary DESC) AS rnk FROM Employees) t WHERE rnk = 2;`

**20. Find records where a group's count exceeds a threshold, but also filter individual rows first.**
Use WHERE for the row-level filter, then GROUP BY, then HAVING for the group-count threshold — same pattern as the WHERE+HAVING scenario covered live today.

**21. Scenario where WHERE and HAVING are both needed.**
*"Total sales by region for Delivered orders only, showing only regions with sales above ₹1,00,000."* WHERE excludes non-Delivered rows before aggregation; HAVING filters the aggregated regional totals. (Full explanation in Live Interview Round log below.)

**22. GROUP BY across multiple columns — how does it change granularity?**
`GROUP BY Region, Product` creates one group per unique Region+Product combination, not per Region alone — this is finer granularity. Every additional GROUP BY column narrows the grouping further.

**23. Performance impact of WHERE vs HAVING when both are logically possible?**
WHERE is always more efficient for row-level conditions since it reduces the row count *before* the costlier grouping/aggregation step. Pushing a row-level filter into HAVING forces the engine to aggregate unnecessary rows first, then discard them — wasted work.

**24. Find categories where average price is above the overall average price.**
```sql
SELECT Category, AVG(Price) AS AvgPrice
FROM Products
GROUP BY Category
HAVING AVG(Price) > (SELECT AVG(Price) FROM Products);
```

**25. Debugging "column must appear in GROUP BY" error.**
Check every column in SELECT — any column not wrapped in an aggregate function must be added to GROUP BY, or removed from SELECT, or aggregated (e.g., wrapped in COUNT/MAX) if only a summary value is needed.

### Section C: Scenario Questions (5) — with Answers

31. **Manager wants sales by region, delivered orders only, above 1L threshold — write full query and explain each clause's purpose.**
```sql
SELECT Region, SUM(Amount) AS TotalSales
FROM Sales
WHERE OrderStatus = 'Delivered'   -- row-level filter, runs before grouping
GROUP BY Region                    -- groups rows by region
HAVING SUM(Amount) > 100000        -- group-level filter, runs after aggregation
ORDER BY TotalSales DESC;          -- sorts final result
```

32. **A query using HAVING for a non-aggregate condition runs slow on a large table — why, and how to fix?**
Because HAVING forces the engine to aggregate all rows first, including ones that could've been excluded earlier — wasted computation. Fix: move any row-level (non-aggregate) condition into WHERE so it filters before the GROUP BY step, reducing the data volume the aggregation has to process.

33. **You need unique customer names AND their total order count — DISTINCT or GROUP BY?**
GROUP BY — since a calculation (COUNT) is needed alongside the unique values, DISTINCT alone can't attach an aggregate.
```sql
SELECT CustomerName, COUNT(*) FROM Orders GROUP BY CustomerName;
```

34. **A table has NULLs in the Amount column. Your SUM(Amount) result seems lower than expected — is this a bug?**
No — this is expected behavior. SUM() automatically ignores/skips NULL values rather than treating them as 0 or throwing an error. If NULLs should be treated as 0 for the business logic, use `SUM(ISNULL(Amount,0))` (SQL Server) or `SUM(COALESCE(Amount,0))` (standard SQL) instead.

35. **Explain to a non-technical stakeholder why a query with `WHERE SUM(Amount) > 1000` failed.**
*"SQL processes a query in stages — it filters individual records first, and only afterward groups and totals them. Asking it to filter by a 'total' before it's even calculated the totals doesn't work — that's why we get an error. The fix is using a different clause (HAVING) that's specifically designed to filter after the totals are calculated."*

## 🧾 Cheat Sheet (One-Page Summary)

| Concept | Rule |
|---|---|
| Execution order | FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY |
| WHERE | Filters rows, before aggregation, no aggregate functions allowed |
| HAVING | Filters groups, after aggregation, aggregate functions required |
| GROUP BY | Every non-aggregated SELECT column must be listed here |
| NULL comparison | Use `IS NULL` / `IS NOT NULL`, never `= NULL` |
| NULLs in aggregates | SUM/AVG/COUNT(column) ignore NULLs; COUNT(*) counts all rows |
| DISTINCT | Unique rows, no aggregation |
| GROUP BY (as dedup) | Unique + can attach aggregates (SUM/COUNT/AVG) |
| Alias in WHERE | ❌ Not allowed (WHERE runs before SELECT) |
| Alias in ORDER BY | ✅ Allowed (ORDER BY runs after SELECT) |
| Best practice | Always filter row-level conditions with WHERE first — never push them into HAVING (wastes aggregation work) |
